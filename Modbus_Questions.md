# Modbus 与 RS485 通信深度解析 FAQ

本指南旨在解答在 STM32 串口转 RS485 实验中遇到的关于 Modbus 协议的核心疑问，帮助你从物理层（硬件）过渡到协议层（软件）。

---

### 1. 你怎么知道我对于 Modbus 有想法？
因为在工业控制和嵌入式通信领域，**RS485 与 Modbus 几乎是“连体婴儿”**。
当你开始研究如何通过 RS485 连接多个设备、如何保证数据传输的可靠性、如何标准地读写远程寄存器时，Modbus 是最自然、最成熟的选择。你的硬件实验（MAX3485 + STM32）已经搭建好了完美的物理舞台，Modbus 就是接下来的“剧本”。

---

### 2. Modbus 和 RS485 有什么联动？
简单来说：**RS485 是“公路”，Modbus 是“交通规则”。**

*   **物理层 (RS485)**：负责电平信号。它定义了如何用 A/B 线的电压差表示 0 和 1。它支持长距离和多节点，但它不关心你传的是什么。
*   **协议层 (Modbus)**：负责逻辑和语义。它定义了谁先说话（主从结构）、怎么打包数据（帧结构）、怎么校验对错（CRC）。

**联动点：**
1.  **半双工控制**：Modbus 规定了请求-响应机制。在 RS485 这种半双工总线上，STM32 必须在发送 Modbus 请求前拉高 `EN` 引脚（发送模式），发送完最后一字节后立即拉低 `EN`（接收模式），等待从机响应。
2.  **多机寻址**：RS485 总线上可以挂载多个设备。Modbus 帧的第一个字节就是 **从机地址 (Slave ID)**，这使得总线上只有被选中的设备会处理数据，其他设备保持静默。

---

### 3. Modbus RTU 又是啥？

Modbus 有几种“方言”，**RTU (Remote Terminal Unit)** 是在串行链路（RS485/RS232）上最常用的一种。

*   **特点**：数据以紧凑的**二进制**形式传输，效率高。
*   **帧结构**：
    `[设备地址] [功能码] [数据内容] [CRC校验]`
    *   例如：`01 03 00 00 00 01 84 0A` （读取 1 号设备从 0000 开始的 1 个寄存器）。
*   **关键点 (RTU 独有)**：**3.5 字符时间间隔**。Modbus RTU 不使用特定的起始或结束字符，而是靠“静默时间”来区分一帧数据的开始和结束。如果总线上超过 3.5 个字符的时间没有数据，设备就认为当前帧结束。

---

### 4. 常见的功能码有哪些？
虽然 Modbus 协议很长，但 90% 的场景只用到这几个：

| 功能码 | 作用 | 对应操作 |
| :--- | :--- | :--- |
| **01 (0x01)** | 读线圈 | 检查继电器或开关状态 |
| **03 (0x03)** | 读保持寄存器 | 读取传感器数值、配置参数 |
| **05 (0x05)** | 写单个线圈 | 控制单个继电器开关 |
| **06 (0x06)** | 写单个寄存器 | 修改某一个配置参数 |
| **16 (0x10)** | 写多个寄存器 | 一次性设置多个参数 |

---

### 5. 在 STM32 上实现 Modbus 的核心难点？
在你目前的实验基础上，要实现 Modbus 需要注意：

1.  **精确的发送/接收切换**：必须确保 UART 的 `TC`（发送完成）标志触发后再拉低 `EN` 引脚，否则最后一字节会被截断。
2.  **空闲中断 (`IDLE IT`)**：利用 STM32 的串口空闲中断可以完美匹配 Modbus RTU 的 3.5 字符间隔，用来识别一帧数据的结束。
3.  **CRC 校验**：Modbus 需要 16 位的循环冗余校验。你需要写一个 CRC16 函数来计算并验证收发的数据，确保传输过程中没有错码。

---

### 6. 接下来我该做什么？
如果你想在项目中跑通 Modbus：
1.  **尝试解析**：在 `UART_ReceiveToIdle` 回调中，判断收到的第一个字节是否为预设的地址。
2.  **计算校验**：对收到的数据运行 CRC16 算法，看是否与帧末尾的两字节匹配。
3.  **构造回复**：如果地址和校验都对，根据功能码构造一个响应帧发回去。

---

### 7. 既然只是读温湿度，我直接通过串口发字符串（如 "Temp: 25C"）不行吗？为什么要用 Modbus？
完全可以！如果你只是为了“看一眼数据”，直接发字符串最简单。但引入 Modbus 是为了解决**工程化**和**标准化**的问题：

1.  **解决“多机抢话”问题 (核心原因)**：RS485 是一根总线。如果不使用协议，多个节点同时发送字符串，数据会相互碰撞变成乱码。Modbus 规定了**“主站不问，从站不答”**的轮询机制，彻底解决了多节点冲突。
2.  **与工业软件无缝对接**：各种工业组态软件（如组态王）、触摸屏、PLC 都原生支持 Modbus。如果你用 Modbus，你的传感器就能直接接入这些系统，而不需要别人专门写代码来解析你的自定义字符串。
3.  **防错能力**：长距离传输容易受干扰。Modbus 强制携带 **CRC16 校验**，保证收到的数据绝对正确，而普通的字符串传输往往没有这么严谨的校验。
4.  **数据结构化**：Modbus 将所有数据抽象为“寄存器”（如温度在 0x01，湿度在 0x02），这种结构极易扩展。

---

### 8. 温湿度传感器对比与选择：DHT11 够用吗？
既然你手头只有 DHT11，**完全可以用它来做初步的实验**。虽然它稳定性差、精度低，但对于验证“STM32 -> MAX3485 -> Modbus”这条通信链路来说，数据来源是 DHT11 还是 SHT20 并不重要。

当你准备升级硬件时，可以参考以下对比：

| 传感器型号 | 接口类型 | 精度与稳定性 | 适用场景 | 备注 |
| :--- | :--- | :--- | :--- | :--- |
| **DHT11** | 单总线 (1-Wire) | 极差，易受干扰，响应慢 | 入门学习、极其粗糙的测试 | 代码稍微复杂（对时序要求极高），不推荐用于实际项目。 |
| **SHT20** | I2C | 较好，工业级芯片 | 室内环境监测、气象站 | 瑞士 Sensirion 出品，性价比高，极其稳定，代码实现（I2C）比单总线更容易。 |
| **SHT30** | I2C | 极好，高精度 | 严苛工业环境、高精度需求 | SHT20 的升级版，响应更快，精度更高，抗干扰能力更强。 |
| **AHT20** | I2C | 好，国产替代佳品 | 消费电子、智能家居 | 国产芯片，价格便宜，性能接近 SHT20，性价比极高。 |

**强烈建议**：后续实验可以使用 I2C 接口的传感器（如 SHT20/AHT20），因为 STM32 的硬件 I2C 或者软件模拟 I2C 比单总线的时序容错率更高。

### 9. RS485 是如何挂载多个单片机的？（硬件拓扑）
RS485 之所以能挂载多个设备，是因为它的物理连接方式是**“手拉手”（Daisy Chain / 菊花链）**拓扑结构，而不是星型或树型。

**硬件接线图解：**
*   **总线 A (Data+)**：所有模块的 A 引脚全部连在一起。
*   **总线 B (Data-)**：所有模块的 B 引脚全部连在一起。
*   **GND**：为了稳定，强烈建议将所有模块的 GND 连在一起（特别是长距离或供电不同的情况下）。

```text
主机 (PC/PLC)
  |-- A -------------------- A --------- A --------- A (总线末端)
  |-- B -------------------- B --------- B --------- B (总线末端)
  |-- GND ------------------ GND ------- GND ------- GND
      |                      |           |           |
 [MAX3485_1]            [MAX3485_2] [MAX3485_3] [MAX3485_4]
      |                      |           |           |
  [STM32_1]              [STM32_2]   [STM32_3]   [STM32_4]
(地址: 0x01)           (地址: 0x02) (地址: 0x03) (地址: 0x04)
```

**关键规则：**
1.  **不能有分叉（避免星型）**：总线必须像一条长绳子一样，从头拉到尾。尽量避免长长的分支（Stub）。
2.  **终端电阻**：如果在长距离（数百米）传输且波特率较高时，为了防止信号反射，通常需要在总线的**最开头和最末尾**（且仅在这两处）各并联一个 **120 欧姆** 的电阻（接在 A 和 B 之间）。
3.  **地址唯一**：在软件层（Modbus），挂在同一条线上的每个 STM32 必须被赋予**不同的设备地址**（如 1, 2, 3...），这样主机喊谁，谁才回答。
